Свод законов проекта
===================
Имя репозитория и кода: model (корневой каталог в Git). Названия «Мастер», «Проект Мастер» в канонических PDF и мастер-тексте в private/ — про продуктовую идею и исходные документы, не про имя этого репозитория.

Формат: фиксируем только подтверждённые тобой решения. Позже приведём к более жёсткой структуре (нумерация, разделы, версии).

Закон 1 — Личный текст только у владельца (Git)
----------------------------------------------
Смысл: файл private/USER_PRIVATE_TEXT.txt и каталог private/ (кроме .gitkeep) не коммитить, не публиковать, не тащить в артефакты.
Подробности и формулировки: PROJECT_LAW.txt.

Закон 2 — Свод законов
----------------------
Вести этот файл как единый реестр утверждённых «законов». К ним возвращаемся при развитии проекта; структура может уточняться позже.

Закон 3 — Документация подтверждённого
--------------------------------------
Фиксировать в репозитории всё, что ты явно подтвердил как канон: решения, ограничения, структуры из промптов (в форме, уместной для текущего этапа). Канонические тексты в Git: (2) `CANON_SECONDARY_MASTER.txt` — извлечение из PDF «Мастер как многослойная private social operating system»; (3) `CANON_PROJECT_MASTER.txt` — извлечение из PDF «Проект Мастер». Иерархия: выше них — `SWOD_ZAKONOV.txt`; мастер-намерение — в `private/` по закону 11; при противоречии — законы SWOD и явные указания владельца. Операционный чеклист разработки вести в `PLAN_REALIZATSII.md` (корень репозитория).

Закон 4 — Новые документы только после согласования
---------------------------------------------------
Перед созданием нового файла документации (новый README, отдельный гайд, второй свод и т.п.) — сначала спросить у тебя, нужен ли именно такой документ. Это обязательное правило процесса.

Закон 5 — Агенты и вспомогательные процессы
------------------------------------------
Когда это ускоряет или повышает качество (разведка по кодовой базе, длинные проверки, параллельные задачи), подключать вложенных агентов и автоматизацию без стеснения — в рамках безопасности репозитория и закона 1 (без утечки private/).

Закон 6 — Локальный веб-интерфейс структурирования потока
---------------------------------------------------------
Цель: глубокий разбор «потока сознания» и удобная структуризация через браузер на локальной машине.

Стек (каталог web/):
- Node.js LTS-уровень: в корне репозитория файл .nvmrc → 22 (через nvm: PATH …/v22.x.x/bin).
- Vite 8 + React 19 + TypeScript 6 — быстрая сборка и HMR.
- Tailwind CSS 4 (@tailwindcss/vite) — стили.
- react-router-dom 7 — маршрутизация.
- zustand (+ persist) — состояние рабочего места и анализа в localStorage браузера.
- @fontsource-variable/inter — шрифт без внешних CDN.

Команды:
- Установка: cd web && npm install
- Разработка (localhost, доступ по LAN при необходимости): cd web && npm run dev — в терминале будет URL (по умолчанию http://localhost:5173).
- Прод-сборка: cd web && npm run build; предпросмотр: npm run preview.

Ожидание после мастер-промпта с текстом: развернутый в браузере интерфейс рабочего места с подключённым анализом и визуализацией (уточняется по содержанию текста).

Закон 7 — Dev-сервер под управлением ассистента
-----------------------------------------------
Ассистент сам поднимает локальный dev-сервер каталога web/ (Vite) при работе над интерфейсом и сам перезапускает его, когда это требуется (смена конфигурации/порта, подозрение на зависание, необходимость чистого старта после правок), без ожидания отдельной просьбы пользователя открыть терминал.

Закон 8 — Структуры из всех промптов
------------------------------------
Все структуры и требования, явно выделенные в промптах пользователя, учитывать и накапливать: в репозитории (свод, код, сиды данных) и в веб-интерфейсе на отдельной вкладке «Структуры из промптов» для последующего разбора. Новые элементы добавлять по мере подтверждённых сообщений.

Закон 9 — Повтор алгоритма и выделение метода
---------------------------------------------
Если один и тот же алгоритм (одна и та же логика) повторяется больше двух раз — обязательно выносить её в отдельную именованную функцию или метод с кратким описанием. Относится к коду проекта и к паттернам обработки текста, где это уместно.

Закон 10 — Фундаментальный текст и глубина анализа
--------------------------------------------------
Фундаментальные тексты и аналогичные файлы (в т.ч. private/USER_PRIVATE_TEXT.txt, когда его содержимое переносится в рабочее место) разбирать максимально глубоко: подключать самые сильные модели/агенты и многослойные разборы. Результаты отражать в веб-интерфейсе как структуры (локальный конвейер + импорт JSON внешнего анализа) для дальнейшей корректировки и исследования; инструменты поддерживать в каталоге web/ (модуль web/src/analysis и связанные сторы/экраны).

Закон 11 — Мастер-текст (откровение и закон намерения)
------------------------------------------------------
Мастер-текст потока, объявленный владельцем откровением и законом намерения для продукта, хранить только локально (например private/MASTER_TEXT.txt); в Git не коммитить. При противоречии трактовок в разработке и документации приоритет у этого слоя до явного обновления или отмены владельцем. Глубокие структурированные выводы по нему — в UI (импорт JSON анализа) и при необходимости в своде на уровне принципов, без публикации сырого текста.

Закон 12 — Режим «Оркестр» в веб-интерфейсе
-------------------------------------------
В каталоге web/ поддерживать вкладку «Оркестр» (/orchestra): планирование ролей вложенных агентов (по типам Cursor и др.), задач дорожек, статусов и заметок дирижёра; персистентность в браузере (ключ localStorage model-orchestra-v1); экспорт и импорт JSON для передачи контекста ассистенту. Сам UI не запускает внешних агентов — только фиксирует план параллельной работы в духе закона 5.

Закон 13 — Продуктовая рамка и монетизация (стратегический канон чата)
-----------------------------------------------------------------------
Смысл: единый опорный текст по позиции продукта и деньгам, чтобы ассистенты и люди не спорили с «памятью чата»; дополняется только по твоему явному подтверждению.

Позиционирование (голубой океан):
- Не борьба за узкие ниши «как у всех», а слой для любого, у кого есть смартфон или компьютер: один каркас, куда каждый подключает нужное.
- Пользователь настраивает интерфейс под себя; на глубине — по смыслу как у разработчика: доступ к кодовой структуре своего репозитория, свои скиллы и связки.
- Удачные пользовательские решения забираются в экосистему; собственные решения команды тоже поставляются и настраиваются как скиллы — у кого скилл установлен, тот с ним взаимодействует.

Иллюстративные сегменты (без фиксации на одном «пляже»; подъезд и прочее — только пример широты аудитории, не обязательная вертикаль):
- Предприниматели: личная CRM + глубоко интегрированный рабочий слой.
- Дом / семья: домашние дела, календари, напоминания, совместные планы, цели (структурно и на автомате).
- Сообщества / локальные чаты: структура общения, расписания, встречи, сборы, планы (класс задач, не единственный фокус).
- Маркетологи / аналитики: сбор и структурирование из разных источников, анализ проектов, графики, таблицы, отчётность.
- Канал: продавцы / внедренцы — сервис может быть бесплатным конечному пользователю, заработок на внедрении и сопровождении (модель дистрибуции; не смешивать без нужды с «семейным» питчем в одном абзаце).

Монетизация (утверждено):
- Гибрид: база доступна широко; то, что тянет дорогие ресурсы — платно.
- Точно платное: общение с внутренним агентом и всё, что расходует ресурсы и токены.
- На старте в голове: пользователь платит за пакет (лимиты/уровень), а не «за каждый токен в лоб» в интерфейсе.
- В перспективе две линии монетизации (детали — после твоего решения и правки этого закона): например подписка/продукт + отдельная линия на агента/пополнение.

Сверка ассистентами: закон 13 (и конспект `docs/STRATEGY_KONSPEKT.md`) читать **только** если в поручении или промпте для агента есть явное указание; не подмешивать в каждую техническую задачу по умолчанию.

Закон 14 — Чат Философа, конспекты и промпты для исполнителей
--------------------------------------------------------------
Роли:
- Чат с агентом **Философ** — психология, философия, смысл, стратегия, приоритеты; **не** исполнение кода, деплоя и правок в репозитории.
- **Исполнители** (Дас, UI, инфра и др.) — делают по **отдельному** промпту с конкретным заданием, который владелец формирует здесь или копирует из конспекта.

Фиксация (не транскрипт чата):
- В репозиторий попадает **только суть** подтверждённого: короткие тезисы в SWOD, развёрнутый конспект для промптов — в `docs/STRATEGY_KONSPEKT.md` (и при необходимости другие файлы в `docs/` **после** твоего согласия на новый документ).
- Психология и личная философия владельца — **только локально** `private/PHILOSOPHY_OWNER.txt` (закон 1); правка в UI: **Мастер-админ → Философия** (`/master-admin/philosophy`), синхронизация с диском при `npm run dev` в `web/`.
- **Карта векторов** — сиды `web/src/data/philosophyVectorsSeed.ts`: продукт, процесс, операции, личные файлы и **слой «Жизнь»** (кто ты, цели, роли, фокус; содержание — в `PHILOSOPHY_OWNER.txt` / чат, не выдумка модели); синхронизировать с репозиторием после существенных сдвигов; обновлять после «фиксируем».
- Полный лог чата **не** переносить в Git и не вставлять в промпты исполнителям целиком.

Промпты для других агентов:
- Из этого чата — **выжимка** (цель, границы, что не делать, пути, критерий готово) + ссылка на нужный закон/конспект **только если** это нужно для задачи.
- В промпте исполнителю явно писать, **что прочитать** (например: «сверься с законом 13» / «не читай SWOD»); без указания — агент **не обязан** открывать SWOD, каноны и конспекты.

Обновление: Философ (или владелец) дополняет конспект и SWOD после твоего «да, фиксируем»; спорные трактовки — спросить, не додумывать. По твоему постоянному поручению «переноси сам» агент **Философ** после существенного разговора о слое **«Жизнь»** сам переносит выжимку в `private/PHILOSOPHY_OWNER.txt` и краткие тезисы — в `web/src/data/philosophyVectorsSeed.ts`, затем при изменении UI — **закон 15** (деплой); сокровенные детали в Git не тащить.

Закон 15 — Админка и UI: владелец всегда видит изменения глазами
------------------------------------------------------------------
Смысл: правки в интерфейсе (Мастер-админ, `/admin`, маршруты и вкладки в `web/`) **не считаются готовыми**, пока ты **не увидел** результат в браузере.

Обязательный порядок для ассистента после изменений UI:
1. `cd web && npm run build` (или `npm run dev`, если договорились только локально).
2. **Прод (основной контур для глаз):** выкладка `web/dist` на VPS **104.171.141.49** → `/var/www/model/web/` (см. `docs/environment/DEPLOY.md`), затем `nginx reload`.
3. Сообщить тебе **прямой URL** проверки (например `http://104.171.141.49/master-admin/philosophy`) и напомнить про жёсткое обновление страницы при кэше.
4. Задача по UI **не закрывается** формулировкой «сделано в коде» без шага 2–3, если ты не сказал явно «только локально, на сервер не выкатывать».

Локальный dev (`localhost:5173`) — допустим **дополнительно**, но не заменяет прод, если ты смотришь админку по IP/домену.
